* Goal
Build an OAuth 2.1 authorization server (with PKCE, Dynamic Client Registration, and OAuth metadata discovery) to front a Streamable-HTTP Bluebeam MCP server, enabling Atlassian Rovo to authenticate and access the MCP server's tools.

[[https://support.atlassian.com/organization-administration/docs/add-an-external-mcp-server-from-atlassian-administration/][Atlassian Oauth Server Requirements]]

* Architecture Overview
- *Authorization Server (AS):* Keycloak, self-hosted on Raspberry Pi 5 (8GB), at oauth-serv.wegwil.com
- *Resource Server (RS):* Existing Node.js Bluebeam MCP server at bb-mcp.wegwil.com
- *Issuer URL (PERMANENT):* https://oauth-serv.wegwil.com/realms/bluebeam-mcp
- *Flow:* Rovo → 401 challenge from RS → discovers AS via protected-resource metadata → DCR self-registration → auth code + PKCE → token → authenticated MCP calls

That issuer string ends up in every token and in your discovery metadata, so once Atlassian registers against it, you don't want to change the hostname or realm name. Just good to know before we pick the realm name in Decision 5.

** Design Decision 2: Ingress

Add to the cloudflared tunnel config.yml
#+begin_src yaml
ingress:
  - hostname: oauth-serv.wegwil.com
    service: http://localhost:8080      # Keycloak's HTTP port
  - service: http_status:404

#+end_src

#+begin_src vbnet
Ingress:  Cloudflare Tunnel (cloudflared on Pi 5)
          oauth-serv.wegwil.com → localhost:8080 (Keycloak)
          bb-mcp.wegwil.com     → localhost:<mcp-port> (existing)
TLS:      Terminated at Cloudflare edge
Keycloak: Configured for proxy/edge TLS (KC_PROXY_HEADERS, KC_HOSTNAME)

#+end_src

Two important notes specific to Keycloak behind a tunnel:

1. TLS terminates at Cloudflare, not on the Pi. The tunnel hands Keycloak plain HTTP on localhost. Keycloak must be told it's running behind a proxy that terminates TLS, otherwise it will generate http:// URLs in its discovery metadata and tokens, or reject requests. You'll set:

    KC_PROXY_HEADERS=xforwarded
    KC_HOSTNAME=https://oauth-serv.wegwil.com
    KC_HTTP_ENABLED=true

This makes Keycloak trust the forwarded headers and emit correct https://oauth-serv.wegwil.com/... issuer and endpoint URLs even though it's serving plain HTTP locally.

2. Cloudflare proxy settings. Make sure the oauth-serv hostname is proxied (orange cloud) like your MCP one. Watch out for any Cloudflare features that could interfere with OAuth — caching should not touch /.well-known/*, /realms/*/protocol/*, or /token-type endpoints. Tunnels generally don't cache by default, but if you have aggressive page rules or "Cache Everything," exclude these paths.

#+end_src

Then create a database and user for Keycloak. This is just normal Postgres admin, nothing exotic.

3. Keycloak — download the distribution (a .tar.gz), unpack it to something like /opt/keycloak, configure it, and run it as a systemd service. Keycloak ships as a plain Java app — no Docker required.

4. systemd unit — a service file pointing at Keycloak's kc.sh start, with your config as environment variables or in conf/keycloak.conf. Exactly the pattern you already use for your other services.

Why not skip Postgres and use the embedded DB?
You'll see Keycloak can run with an embedded H2 database with zero setup. Resist this. H2 is for dev only — it can corrupt, doesn't handle restarts gracefully, and Keycloak explicitly warns against it for real use. Since Atlassian will register a client and users will authenticate, you need that data to survive reboots. Postgres is worth the modest extra setup, and you already know apt.

** Design Decision 6: Dynamic Client Registration (DCR) Policy
This is the make-or-break decision. Atlassian needs to self-register a client at your registration endpoint with no pre-shared credential, but an open registration endpoint is a genuine attack surface. So the design goal is: allow Atlassian's anonymous registration while constraining what a registered client can do.

Here's the landscape and the decisions within it.
The core tension

- Keycloak default: anonymous DCR is effectively locked down — registration typically requires an initial access token that you generate manually. But Atlassian won't have one; it expects to POST to the registration endpoint unauthenticated.
- What you must do: permit anonymous registration on the bluebeam-mcp realm, then lean on Keycloak's Client Registration Policies to keep that safe.

The registration endpoint

Keycloak exposes OIDC Dynamic Client Registration at:
#+begin_src perl
https://oauth-serv.wegwil.com/realms/bluebeam-mcp/clients-registrations/openid-connect

#+end_src

One thing to verify during testing: the path Atlassian discovers via your metadata must point here. The MCP/OAuth world expects a registration_endpoint in the discovery document, and Keycloak advertises this path in its .well-known. We'll confirm the discovery doc lines up when we test.
Sub-decision 6a: How permissive is anonymous registration?

*** Sub-decision 6a: How permissive is anonymous registration? 
  Keycloak applies Client Registration Policies to anonymous registrations. The key ones:

  - Trusted Hosts — by default this can restrict which hosts may register or which redirect URIs/host URLs are allowed. This is your strongest lever. Atlassian's registration call and redirect URIs come from Atlassian's infrastructure, so you could restrict to their domains — but their IPs/hosts can change, so this is a tradeoff between security and brittleness.
  - Consent Required — can force user consent on registered clients.
  - Full Scope Disabled — prevents registered clients from getting broad role mappings. Good to keep on.
  - Allowed Client Scopes / Protocol Mappers — constrain what a self-registered client can request.

Decision: Start with anonymous DCR enabled but with the default protective policies left on (Full Scope Disabled, Allowed Protocol Mappers, Allowed Client Scopes). Get the integration working first, then optionally tighten Trusted Hosts to Atlassian's domains once you know exactly what hostnames/redirect URIs they use. Trying to lock down Trusted Hosts before you've seen a successful registration tends to cause silent failures that are hard to diagnose.

*** Sub-decision 6b: PKCE enforcement

  OAuth 2.1 / Atlassian require PKCE with S256. A DCR-registered client won't necessarily have PKCE enforced unless you make it the realm/client default. 
  We'll set the client to require S256. This can be applied via a default for registered clients or verified/patched on the registered client after the fact.

*** Sub-decision 6c: Mitigating the open endpoint
  Since the endpoint is anonymous, add defense-in-depth:

    Cloudflare rate limiting on the clients-registrations path (you already have Cloudflare in front — this is an easy win).
    Optionally a Cloudflare WAF rule restricting that path to expected sources later.
    Keep the protective registration policies enabled.

    #+begin_src yml
DCR:            Anonymous registration ENABLED on realm bluebeam-mcp
Endpoint:       /realms/bluebeam-mcp/clients-registrations/openid-connect
Policies:       Keep default protective policies ON
                (Full Scope Disabled, Allowed Client Scopes, Allowed Protocol Mappers)
Trusted Hosts:  Leave permissive initially; tighten to Atlassian domains AFTER first success
PKCE:           Enforce S256 on registered client(s)
Hardening:      Cloudflare rate-limit + optional WAF on registration path
Approach:       Get registration working first, harden second

    #+end_src

** Design Decision 7: MCP Resource Server (token validation+metadata)
Your Bluebeam MCP server at bb-mcp.wegwil.com takes on the role of the OAuth Resource Server (RS). Keycloak is the Authorization Server (issues tokens); your MCP server is the Resource Server (validates tokens and serves the protected tools). Keycloak does not do any of the work in this decision — these are responsibilities you add to your Node application.

Responsibility 1 — Protected Resource Metadata (RFC 9728)

Your MCP server must advertise where authentication happens, via a two-part handshake:
The 401 challenge. Any request to your MCP endpoint without a valid token must return HTTP 401 with a WWW-Authenticate header that points to your protected-resource metadata document:

#+begin_src python
WWW-Authenticate: Bearer resource_metadata="https://bb-mcp.wegwil.com/.well-known/oauth-protected-resource"
#+end_src

The metadata document. You serve a JSON file at https://bb-mcp.wegwil.com/.well-known/oauth-protected-resource that names your resource and points at the Keycloak realm as the authorization server:

#+begin_src json
{
  "resource": "https://bb-mcp.wegwil.com",
  "authorization_servers": ["https://oauth-serv.wegwil.com/realms/bluebeam-mcp"]
}

#+end_src

This small file is the bridge between your two domains: Rovo reads authorization_servers from it, then fetches Keycloak's own discovery document to find the authorize, token, and registration endpoints.
Responsibility 2 — Access Token Validation

Once Rovo presents Authorization: Bearer <jwt>, your server validates the token before serving any tool. The required checks are: verify the signature against Keycloak's JWKS (public keys) fetched from https://oauth-serv.wegwil.com/realms/bluebeam-mcp/protocol/openid-connect/certs; confirm iss equals https://oauth-serv.wegwil.com/realms/bluebeam-mcp; confirm aud includes https://bb-mcp.wegwil.com; and confirm the token is within its exp / nbf validity window.
The audience-binding dependency (cross-cutting)

By default Keycloak does not include your MCP server's URL in the token's aud claim. You must add an audience mapper in the bluebeam-mcp realm so issued tokens carry aud: https://bb-mcp.wegwil.com. This is a Keycloak-side config item, but it is the direct counterpart to the aud check above — the mapper's output and your validator's expected audience must match exactly, or every request will fail validation. We deferred the actual mapper configuration to the Keycloak realm-detail step, but it is logically part of this decision.
Sub-decision 7a — Validation approach

We chose local (offline) JWT validation with cached JWKS over token introspection. Your server fetches Keycloak's public keys once, caches them in memory (with automatic handling of key rotation), and validates tokens locally on each request. This is fast and avoids a per-request network call to Keycloak. Introspection (calling Keycloak on every request) was rejected as unnecessary latency and load for this use case.

Sub-decision 7b — Node implementation

Confirmed the MCP server is Node/JavaScript. The chosen libraries are jose for JWT work — specifically createRemoteJWKSet (fetches and caches Keycloak's JWKS with key-rotation handling) and jwtVerify (checks signature, iss, aud, and exp in one call) — while your existing HTTP layer serves the .well-known metadata route and emits the 401 challenge. createRemoteJWKSet's in-memory caching is exactly the offline-validation behavior we wanted.

#+begin_src yaml
Role:               MCP server = OAuth Resource Server
Metadata:           Serve /.well-known/oauth-protected-resource
                    → authorization_servers: [https://oauth-serv.wegwil.com/realms/bluebeam-mcp]
401 challenge:      WWW-Authenticate points to resource_metadata URL
Token validation:   Local JWT validation against Keycloak JWKS (cached in memory)
Checks:             signature, iss, aud=https://bb-mcp.wegwil.com, exp/nbf
Audience binding:   Keycloak audience mapper must emit aud=bb-mcp URL (config'd realm-side)
Validation style:   Offline JWT (not introspection)
Language/libs:      Node / JavaScript, jose (createRemoteJWKSet + jwtVerify)
Open item:          Confirm HTTP framework (Express / Fastify / raw http) at build time
#+end_src>

** Design Decision 8: User Identity
When Rovo redirects a user to Keycloak's /authorize endpoint, someone has to actually log in. This decision is about who those users are and where their accounts live.
The options

A) Keycloak-local users (recommended to start)
You create user accounts directly in the bluebeam-mcp realm in Keycloak. Usernames, passwords, and profiles all live on your Pi in Postgres. No external dependencies — fully self-contained. Best when the user base is small (just you, or a handful of people you manage by hand).

B) Federate to an external IdP (identity brokering)
Keycloak delegates login to a provider users already have — Google, Microsoft/Entra, GitHub, etc. Users click "sign in with Google," authenticate there, and Keycloak brokers that into a token. No password management on your end, but more setup, and you're now dependent on that external provider being configured correctly.

C) LDAP / Active Directory federation
Only relevant if you already run a directory server. Almost certainly not your situation as a home/hobby setup.
Recommendation

For a self-hosted setup on a Pi with a small known set of users, Option A (Keycloak-local users) is the right starting point:

    Nothing external to configure or depend on — keeps the whole system on your Pi.
    One less moving part while you're getting the core OAuth/DCR flow working.
    You can add federation later (Option B) without breaking the Atlassian integration — brokering is additive and doesn't change your issuer, realm, or the tokens' structure from Rovo's perspective.

The main tradeoff: you manage accounts and password resets yourself. For a small user base that's trivial; for dozens of users it'd get tedious, which is when federation starts to pay off.
A security note regardless of option

Since your /authorize page is publicly reachable through the Cloudflare tunnel, consider enabling on the bluebeam-mcp realm:

    A reasonable password policy (length/complexity) for local users.
    Brute-force detection (Keycloak has this built in — Realm Settings → Security Defenses → Brute Force Detection) to lock accounts after repeated failed logins.
    Optionally TOTP/MFA later, since the login page is internet-facing.

    #+begin_src yaml
User source:    Keycloak-local users in realm bluebeam-mcp
Management:      Created/managed by you in the Keycloak admin console
Federation:     None initially (can add Google/Microsoft/etc. later, non-breaking)
Hardening:      Password policy + Brute Force Detection enabled
                MFA/TOTP optional, recommended given internet-facing login

    #+end_src

*** Design Decision 9: Secretes, Backups, and Ops

* Plain Language Implementation Steps/Plan
You're setting up Keycloak — an identity and login server — on your Raspberry Pi 5, and making it securely reachable from the internet. The work breaks into six stages. First you extract the Keycloak software and create a dedicated, unprivileged system user for it to run as, so it isn't running with more power than it needs. Then you configure it: telling it to use a PostgreSQL database, listen only on the Pi's local network interface, and trust the proxy that sits in front of it. Crucially, the passwords and admin credentials live in a separate, locked-down secrets file rather than in the main config, and a systemd service keeps Keycloak running and restarts it if it crashes.

Next you expose it to the world using a Cloudflare Tunnel. Rather than opening ports on your router (which is risky), the tunnel makes an outbound connection from your Pi to Cloudflare, and public traffic flows back through that encrypted channel to Keycloak on localhost. Cloudflare also handles the HTTPS/TLS certificate automatically, so visitors reach auth.example.com over a proper secure connection without you managing certificates yourself.

After it's running and reachable, you harden it. You log in with the temporary bootstrap admin account, create a proper personal admin user, verify it works, and then delete the bootstrap credentials so there's no leftover standing secret. You create a separate "realm" for your actual applications (keeping the master realm purely for administration), register your first app as a client, tune how long login sessions and tokens last, and turn on brute-force protection. Optionally, you can put Cloudflare Access in front of the admin console as a second, independent lock.

Finally, you set up backups and an upgrade procedure — the part most people skip and later regret. A nightly automated job dumps the PostgreSQL database (which holds everything: users, realms, secrets), keeps a rolling set of recent dumps, and copies them onto your external hard drive, encrypted with an age key so the secrets aren't sitting in cleartext. The key safety detail throughout is making sure the drive is genuinely mounted before writing, so a disconnected drive can't silently fill up the Pi's SD card. The upgrade process is deliberately cautious: always back up first, extract the new version alongside the old, swap a symlink, and let Keycloak migrate the database — with the understanding that the only real rollback is restoring that pre-upgrade backup, since schema migrations only go forward.

The end result is a self-hosted, internet-reachable, TLS-secured, brute-force-protected login server with a dedicated app realm, automated encrypted backups to external storage, and a tested path for both restoring and upgrading.

A few things worth flagging before you run it:

The hostname=https://auth.example.com line must match the public hostname your Cloudflare tunnel will serve. Since the tunnel terminates TLS and forwards plain HTTP to 127.0.0.1:8080, the proxy-headers=xforwarded setting tells Keycloak to trust the X-Forwarded-* headers for building correct redirect URLs. Get this wrong and you'll get login loops or "invalid redirect" errors — it's the single most common Keycloak-behind-proxy gotcha.

The bootstrap admin credentials create the temporary initial admin account on first startup. Once you've logged in and created a proper admin user in the master realm, you can remove those two lines from the env file — they're only read when no admin exists yet.

The ProtectSystem=strict plus ReadWritePaths=/opt/keycloak/data hardening means Keycloak can only write to its data directory. If you later add providers or themes that need to write elsewhere, you'll need to add those paths.

     version. This is exactly why the pre-upgrade backup matters.

**** TODO Rollback plan if it goes wrong
     Schema migrations are forward-only — you can't just point the symlink
     back, because the new version may have altered the DB. Rollback means:
     1. Stop keycloak.
     2. Restore the pre-upgrade DB dump (pg_restore --clean).
     3. Point the symlink back at the old version.
     4. Start. This is the only reliable rollback — hence: back up first.
